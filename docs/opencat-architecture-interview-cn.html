OpenCat 项目架构全解（面试版）

1. 项目一句话定位

OpenCat 是一个类 OpenCow 的开源 AI Agent 编排平台，核心目标不是做一个“聊天网页”，而是做一套可自托管、可扩展、可解释的 Agent Runtime：支持多模型接入、Tool Calling、多 Agent 编排、Memory 自动记忆、RAG 知识库、用量统计，以及 Web / Desktop 双端分发。

面试里可以先用一句话定调：

我做的是一个 AI Agent 编排平台，不只是聊天 UI。它更像是把模型、工具、记忆、知识库和多 Agent 协作能力整合成了一套完整运行时。

2. 为什么做这个项目

这个项目的核心价值，不在于“调用了一次大模型 API”，而在于把 AI 应用里最关键的几层都自己串起来：

  • 上层是用户交互和对话体验。
  • 中间层是 Agent Runtime，包括模型选择、工具调用、推理循环、多 Agent 协作。
  • 下层是基础设施，包括 PostgreSQL、pgvector、Redis、鉴权、加密和计费日志。

这个项目故意没有选 LangChain 或 AutoGen 这类重封装框架，原因有三个：

  • 第一，面试时要讲原理，自研 Runtime 更能证明我真正理解 Agent 是怎么跑起来的。
  • 第二，平台需要兼容不同 Provider 和不同 API 协议，自研抽象层更灵活。
  • 第三，很多 AI 框架升级很快、黑盒很多，自己做中间层之后，定位问题和扩展能力都更可控。

一句话总结这个设计思想：

不依赖黑盒框架，自己搭一套可解释、可扩展、可落地的 AI Agent 基础设施。

3. 技术选型与思路

3.1 前端为什么选 Next.js 16

选 Next.js 16 的原因不是为了“追新”，而是因为这个项目同时有页面、API Route、鉴权和服务端逻辑，Next.js 可以把前后端边界收拢在一个仓库里，开发效率高，部署链路也更简单。

这个项目主要使用了 App Router 结构，优点是：

  • 页面路由和接口路由统一管理。
  • 服务端逻辑可以直接写在 API Route 里，适合快速构建 AI 应用。
  • 后续接入 Tauri 桌面端时，Web 代码也能尽量复用。

3.2 为什么用 Prisma 7 + PostgreSQL 17

因为这个项目既需要传统业务数据，又需要向量检索能力。

PostgreSQL 负责存这些结构化数据：

  • 用户、会话、OAuth 账号
  • 项目、Agent、对话、消息
  • API Key、UsageLog
  • 知识库、文档、分块

pgvector 负责存这些向量数据：

  • 用户 Memory 向量
  • 文档 Chunk 向量

Prisma 7 负责 ORM 层，但这里我没有把 Prisma 当成“万能 ORM”。因为 vector 类型 Prisma 还不原生友好，所以我做了分层处理：

  • 普通业务表走 Prisma ORM。
  • 向量写入和相似度检索走原始 SQL。

这其实是一个很典型的工程判断：ORM 提高开发效率，但遇到底层能力缺口时，要敢于回到 SQL。

3.3 为什么还要 Redis

当前版本 Redis 的使用不是最重的，但它是平台型架构必须预留的一层。它的价值主要在：

  • 未来可用于任务队列和异步处理。
  • 可用于热点数据缓存。
  • 复杂 Agent 工作流里，可做状态中转。

所以 Redis 在这里更像是“面向后续扩展预埋的基础设施”。

3.4 为什么选 Vercel AI SDK 6.x

因为它把流式输出、Tool Calling、模型适配这些高频能力统一了，但我没有直接把业务逻辑绑死在 SDK 上，而是在它上面又封了一层 Provider 抽象和 Agent Engine。

也就是说：

  • AI SDK 解决模型调用层问题。
  • 我自己的代码解决业务编排层问题。

这样既吃到了 SDK 的成熟能力，也保留了平台自己的控制权。

4. 整体架构怎么拆

如果面试官让你画架构图，你可以按“四层结构”来讲：

4.1 表现层

这一层就是前端界面，核心是：

  • Chat 对话面板
  • Sidebar 对话列表
  • Settings 配置页
  • Agents 管理页
  • Knowledge Base 管理页
  • Dashboard 统计页

这一层主要负责：

  • 展示消息流
  • 展示工具调用过程
  • 管理模型和 Agent 选择
  • 管理 API Key、知识库和统计数据

4.2 接口层

也就是 Next.js App Router 下的 API Routes，比如：

  • /api/chat
  • /api/conversations
  • /api/keys
  • /api/agents
  • /api/knowledge
  • /api/memory
  • /api/stats

这一层的职责是：

  • 做鉴权
  • 接收前端参数
  • 调用 Runtime
  • 持久化数据
  • 返回流式或结构化结果

4.3 运行时层

这是整个项目最核心的一层，也是面试时最能拉开差距的一层。主要包括：

  • LLM Provider Registry
  • Tool Registry
  • ReAct Agent Engine
  • Orchestrator 多 Agent 协作
  • Memory 检索
  • RAG 检索

这一层是整个平台的大脑，负责决定“用哪个模型、带哪些工具、怎么循环、怎么拿上下文、怎么协作”。

4.4 存储与基础设施层

包括：

  • PostgreSQL 17
  • pgvector
  • Redis 7
  • Prisma 7
  • AES-256-GCM 加密
  • NextAuth v5 鉴权

这一层负责“数据可靠性、安全性和可追踪性”。

5. 数据模型为什么这样设计

这个项目数据库一共设计了 15 张表，核心思路不是“表越多越高级”，而是围绕平台能力做实体拆分。

5.1 用户和认证相关

  • User
  • Account
  • Session
  • VerificationToken

这部分是 NextAuth 的标准认证域，解决登录和 OAuth 问题。

5.2 平台业务核心域

  • Project
  • Agent
  • Conversation
  • Message
  • Tool
  • UsageLog

这里体现的是平台的主业务关系：

  • 一个用户可以有多个项目。
  • 一个项目可以有多个 Agent。
  • 一个对话归属于一个项目，也可以绑定一个 Agent。
  • 每条消息都持久化，方便历史回放和审计。
  • 每次模型调用都会记录 UsageLog，用于成本和数据看板。

5.3 AI 增强能力域

  • Memory
  • KnowledgeBase
  • Document
  • DocumentChunk

这是 AI 平台区别于普通聊天工具的关键：

  • Memory 用来保存长期用户信息。
  • KnowledgeBase 是知识库容器。
  • Document 保存原始文档。
  • DocumentChunk 保存分块文本和向量。

面试里可以这么说：

我的表设计不是围绕页面来切的，而是围绕 Agent 运行时能力来切的，所以能比较自然地支持 Memory、RAG、多 Agent 和用量追踪。

6. 核心链路一：用户发一条消息后，系统怎么跑

这是最重要的一个面试题，你一定要会讲。

完整链路可以拆成 8 步：

第 1 步：前端发起请求

前端通过 useChat 和自定义 transport，把下面这些参数发给 /api/chat：

  • 当前消息
  • conversationId
  • modelId
  • agentId
  • enableTools
  • toolNames

这里的重点是：前端不只是“发个文本”，而是把当前会话上下文和运行配置一起发过去。

第 2 步：服务端校验身份与上下文

/api/chat 先拿当前登录用户，再校验对话是否属于该用户。如果是首次发消息，还会自动创建默认项目和默认对话。

这样做的原因是保证所有数据天然和用户隔离，不会串。

第 3 步：加载 Agent 配置

如果请求里带了 agentId，服务端会把 Agent 的配置取出来，包括：

  • 系统提示词
  • 模型
  • 温度
  • 最大步数
  • 可用工具
  • 是否开启 Orchestrator

这一步决定这次对话到底是“普通聊天”还是“带策略的 Agent 对话”。

第 4 步：注入 Memory 和 RAG 上下文

在真正调用模型之前，系统会做两次召回：

  • 从 Memory 里按语义相似度检索相关记忆。
  • 从知识库分块里按语义相似度检索相关文档片段。

然后把这些内容拼成系统提示词后缀，注入给 Agent。

重点不是“把所有历史都塞给模型”，而是只塞跟当前问题最相关的上下文，这样能兼顾效果和成本。

第 5 步：创建模型实例

系统根据 modelId 和用户自己的 API Key，动态创建模型实例。这里走的是统一 Provider Registry。

也就是说，OpenAI、Anthropic、DeepSeek、Gemini，在业务层看起来都是同一种调用方式。

第 6 步：进入 ReAct Agent Engine

如果开启了工具或 Agent，这次请求就不是一次简单的 generateText，而是进入 ReAct 循环：

  • 模型先思考下一步要不要行动。
  • 如果要行动，就调用某个工具。
  • 拿到工具结果后，再继续推理。
  • 直到达到停止条件。

停止条件不是写死的 while true，而是通过 stopWhen: stepCountIs(N) 控制，防止无限循环。

第 7 步：流式返回结果

服务端通过 AI SDK 的流式响应接口，把结果持续推给前端。前端边收边渲染，用户看到的是类似 ChatGPT 的实时输出体验。

如果过程里有工具调用，前端也会把工具卡片一起渲染出来，让整个链路可见。

第 8 步：异步持久化

最后系统会把这些数据落库：

  • 用户消息
  • AI 回复
  • Token 用量
  • 花费金额
  • Conversation 元数据

这样后续才能支撑：

  • 历史消息回放
  • Dashboard 看板
  • 成本追踪
  • 面向企业场景的审计需求

7. 核心模块一：多模型 Provider 抽象

这个模块的价值在于把“模型接入问题”从业务逻辑里拆出去。

7.1 为什么一定要抽象 Provider 层

如果不抽象，业务代码里很快会充满这种判断：

  • 如果是 OpenAI 就调这个方法
  • 如果是 Anthropic 就改另一个参数
  • 如果是代理平台就换 baseUrl

这样很快就会变成一团条件分支，后面维护很痛苦。

所以我做了 src/lib/llm/ 这层，统一处理：

  • Provider 定义
  • 模型注册
  • 价格配置
  • API 格式差异
  • 模型实例创建

7.2 这个模块解决了什么真实问题

AI SDK 6.x 有一个很现实的坑：OpenAI 官方默认可以走 Responses API，但很多第三方兼容平台其实只支持 Chat Completions。

所以我引入了 ApiFormat 概念，区分：

  • openai
  • openai-responses
  • anthropic
  • google-genai

这样做以后：

  • OpenAI 官方可以走它自己的新协议。
  • DeepSeek 和代理平台走 OpenAI 兼容协议。
  • Google 和 Anthropic 走各自协议。

这不是单纯“为了解耦”，而是为了解决真实兼容问题。

7.3 面试怎么说

我把模型适配做成了注册表 + 工厂模式，业务层只关心 modelId，不关心底层到底是 OpenAI、Anthropic 还是代理平台，这样后续接更多模型只要扩 Provider 层，不需要改聊天主流程。

8. 核心模块二：Tool Calling 与 ReAct Agent Engine

这是项目最核心的技术亮点之一。

8.1 Tool Registry 是干什么的

我先做了 Tool 的类型系统和注册中心，把工具统一成标准结构：

  • 名称
  • 描述
  • 输入 Schema
  • 执行函数
  • 开关状态

这样做的好处是所有工具都能被统一管理，也能统一转给 AI SDK。

目前内置工具包括：

  • calculator
  • datetime
  • http_request
  • memory_save
  • memory_search
  • call_agent

8.2 ReAct Engine 是怎么设计的

我没有把 Agent 理解成“一个模型接口”，而是理解成一个“推理循环控制器”。

核心循环是：

  1. 让模型看到问题、系统提示词和可用工具。
  2. 模型判断是否需要调用工具。
  3. 执行工具，拿到 observation。
  4. 把 observation 回灌给模型继续推理。
  5. 达到停止条件后输出最终答案。

这个模式本质上就是 ReAct，只不过我把它工程化了。

8.3 为什么工具过程要可视化

很多 AI 产品的一个问题是：结果有了，但过程完全黑盒。

我这里把工具调用状态也渲染到了消息流里，能看到：

  • 调用了哪个工具
  • 传了什么参数
  • 执行成功还是失败
  • 返回了什么结果

对用户来说这是可解释性，对开发者来说这是调试能力。

9. 核心模块三：多 Agent 编排为什么不是噱头

很多项目写“多 Agent”，其实只是多个 prompt 模板。这个项目不是这么做的。

9.1 我的 Orchestrator 设计

我把子 Agent 调用做成了一个真实工具：call_agent。

也就是说，主 Agent 并不是硬编码“先调 A 再调 B”，而是由 Orchestrator 根据当前任务判断：

  • 要不要拆分任务
  • 调哪个子 Agent
  • 子 Agent 的输出怎么再汇总

这就形成了一个比较真实的多 Agent 调度模式。

9.2 这种设计的优点

  • 扩展性好，后续新增子 Agent 不需要重写主流程。
  • 行为更自然，主 Agent 可以动态决策。
  • 能把复杂任务拆成更稳定的小任务。

面试里可以这么说：

我把多 Agent 协作建模成一种 Tool Calling，而不是单独再造一套协议。这样 Orchestrator 和普通工具共享一套机制，系统复杂度会低很多。

这句话很加分，因为它说明你不是只会堆概念，而是会控制系统复杂度。

10. 核心模块四：Memory 和 RAG 为什么要分开

这是很多面试官会问的点。

10.1 Memory 解决什么问题

Memory 解决的是“关于用户本身的长期信息”，例如：

  • 用户偏好什么风格
  • 用户过去提过哪些固定背景
  • 用户习惯的输出方式

它更像长期个性化记忆。

10.2 RAG 解决什么问题

RAG 解决的是“外部知识上下文补充”，例如：

  • 用户上传的产品文档
  • 内部 SOP
  • 项目说明文档

它不是在记用户，而是在记知识资料。

10.3 为什么不能混成一套

因为它们的数据来源、生命周期和用途都不同：

  • Memory 是对用户的长期抽象。
  • RAG 是对文档内容的检索增强。

如果混在一起，会出现两个问题：

  • 用户长期偏好和知识文档互相污染。
  • 检索策略很难调优。

所以我把它们拆成两套，但在对话入口统一注入。

这就是架构上的“分而治之，入口汇总”。

10.4 RAG 管线是怎么做的

文档上传后会经历：

  1. 文本解析
  2. 滑动窗口分块
  3. embedding 批量向量化
  4. 写入 DocumentChunk
  5. 查询时做向量召回

这里我专门做了句子边界感知和 chunk overlap，原因是纯定长切分很容易把语义切断，检索质量会下降。

11. 安全与工程化考虑

这个项目虽然是 7 天完成，但不是 Demo 思维，而是尽量按生产思维做。

11.1 API Key 为什么要 AES-256-GCM

用户自定义 Provider 的 API Key 是平台的高敏感信息，不能明文存库。

所以我做了专门的加密模块，存储时加密、使用时解密，数据库里永远不落明文。

这里选 AES-256-GCM 的原因是：

  • 强度足够
  • 带认证能力
  • 工程落地成熟

11.2 为什么要记录 UsageLog

因为 AI 平台如果没有用量日志，后面很多功能根本做不起来：

  • 成本统计
  • 模型对比
  • 用户配额
  • Dashboard 趋势分析
  • 企业计费

所以 UsageLog 不是“顺手记一下”，而是平台商业化的基础。

11.3 为什么做 Dashboard

Dashboard 的意义不是好看，而是验证平台有没有从“能用”进入“可运营”。

它反映的是三个能力：

  • 数据被完整记录了
  • 数据能被正确聚合
  • 数据能被用户看懂

所以我把 /api/stats 聚合做成单接口，并用原生 SVG 自己画图，不依赖重图表库，控制包体和可维护性。

12. 为什么还能做成桌面端

这个点在面试里也很好用，因为它说明项目不是单一交付形态。

我选 Tauri 2.0 而不是 Electron，主要原因是：

  • 安装包更小
  • 资源占用更低
  • Rust 底层对简历也更加分

而且这个项目本身前端和接口边界清晰，所以 Web 版和 Desktop 版可以复用绝大多数代码。

一句话总结：

我不是额外重写了一套桌面应用，而是把 Web 架构设计成天然适合二次封装为 Desktop Client。

13. 这个项目最有价值的面试亮点

如果面试官问“你觉得这个项目最体现你能力的地方是什么”，你可以说下面 5 点：

亮点 1：不是简单接 API，而是自己搭了 Runtime

说明你不只是会调模型，而是理解 AI 应用中间层。

亮点 2：Provider 抽象做得比较完整

说明你有平台思维，不是只会写单点业务。

亮点 3：ReAct + Tool Calling + 多 Agent 是一套统一机制

说明你能控制复杂度，能把概念整合成工程实现。

亮点 4：Memory 和 RAG 做了明确边界

说明你理解 AI 增强能力在数据层面的区别。

亮点 5：有安全、成本、运营这些工程化设计

说明你不是做玩具项目，而是按真正产品思维在做。

14. 常见面试问题与推荐回答

14.1 为什么不用 LangChain

推荐回答：

我不是排斥 LangChain，而是这个项目的目标之一就是把 Agent Runtime 原理讲清楚、做扎实。自己实现 Provider Registry、Tool Registry、ReAct Loop 和 Memory/RAG 注入之后，后续不管是接 LangChain 还是继续自研，我都会更有控制力。

14.2 为什么 Memory 和 RAG 要分开

推荐回答：

Memory 记的是用户长期信息，RAG 检索的是外部知识文档，这两个东西的数据来源和使用场景完全不一样。拆开之后检索质量和系统边界都会更清晰。

14.3 多 Agent 的价值是什么

推荐回答：

复杂任务如果全压给一个 Agent，稳定性和可控性都差。多 Agent 的核心不是数量多，而是任务分工清晰。我的做法是让主 Agent 通过 call_agent 动态调子 Agent，本质上还是统一到 Tool Calling 机制里。

14.4 为什么日志和成本统计这么早就做

推荐回答：

OpenCat 是我做的一个开源 AI Agent 编排平台，定位不是普通聊天应用，而是一套可自托管的 Agent Runtime。技术上我用 Next.js 16 做前后端一体化开发，Prisma 7 + PostgreSQL 17 负责业务数据，pgvector 负责 Memory 和 RAG 的向量检索，Redis 作为后续异步和缓存基础设施。
